# Agent Driver Spec (2026-03-18)

本文档作为后续调度器修复的驱动规范。代码行为以本文档为准。

## 1. 总体目标

- 调度流程必须是 `先快速收敛，再基于 token drift(JSD) 做局部微调`。
- 禁止在早期阶段把系统长期留在极端 workload。
- 禁止在未收敛前就用 JSD 驱动频繁局部扰动。

## 2. 四阶段执行流

### Phase A: Warmup (1-2 窗)

- 保持初始配置不变。
- 不进行 seek、reroute、whitelist break。
- 目的：建立初始观测基线。

### Phase B: 快速收敛段 (最多 10-12 窗)

- 允许大步迁移：单步可用 `|delta ctx|<=2`、`|delta inf|<=2`。
- 目标：把极大或极小 workload 拉回正常区间（例如向 `(2,2)` 附近收敛）。
- 该阶段允许探索与回退。
- 大 workload 优先下压探索。
- 极小 workload 优先上探探索。
- 退出条件：
- 达到目标区间且局部稳定。
- 或达到最大窗口数上限。

### Phase C: 稳定确认段 (1-2 窗)

- 暂停切换，固定当前候选优点。
- 不允许 whitelist 大幅 break。
- 目的：验证快速收敛结果是否可持续。

### Phase D: JSD 驱动局部微调

- 仅在 Phase C 完成后启用 token drift(JSD) 驱动。
- 只做局部探索，使用有限半径与有限窗口。
- 安全约束可短暂局部放开，但必须受窗口和半径约束。
- 不再执行全局大步搜索。

## 3. 关键行为约束

- `cold_start` 的职责是快速收敛，不是长期反复探测。
- `whitelist_break` 只能服务于收敛，不得在 warmup 或稳定确认段触发。
- 对 `(1,1)` 这类极小点，允许短暂上探，但必须在收敛窗口内明确回落到稳定点。
- 对 `(8,1)/(1,8)/(8,2)/(2,8)` 这类大点，必须出现明显下压轨迹。

## 4. 验收标准

- 大 workload 轨迹应呈现“有限探索 + 快速下压 + 稳定确认”。
- 调度日志应能清晰分辨四阶段，不混叠。
- JSD 触发只在后期局部微调段生效。
- 不应再出现“早期长时间 seek 抖动但无净收敛”。

## 5. 实施原则

- 采用最小侵入改动，不破坏已有可用路径。
- 优先改阶段门控和触发时序，再调参数细节。
- 每轮改动都要配套导出调度序列进行阶段一致性检查。
